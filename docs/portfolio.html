01 — 문제

바이브코딩 게임 생성의 두 구멍

AI로 게임을 만들 때 두 가지가 늘 막혀요. 하나, 과정이 안 보여요 — 어디서 막혔는지, AI가 실제로 뭘 했는지, 만든 코드가 진짜 도는지 알 수 없어요. 둘, 매번 처음부터 짜요 — 어제 만든 걸 오늘이 재활용하지 못해요. AgentForge는 생성을 단계로 쪼개 들여다보고, 바꾸기 전에 플레이 가능성을 증명하고, 지을수록 메모리가 쌓이는 “통제된·진화하는 생산”을 지향해요.

02 — 어떻게 도나

Codex CLI 5단계 파이프라인

npm run generate -- "<아이디어>" 한 줄이면, Node 오케스트레이터가 Codex CLI(codex exec --json · 내장 image_gen)를 단계별로 불러요. 코드와 이미지가 한 흐름이고(브라우저 직접호출은 폐기했어요), 결과는 runs/<id>/에 그대로 남아요.

출처: orchestrator/{codex-cli·generate·template-api·reachability·evolution}.ts · ③Gate 재현 6b58e1f / 3658820 · [사실, 실재]
03 — 설계 결정

라벨이 아니라, 작동하는 코드로

vault에 설계로만 있던 게임 스튜디오 이론(자가진화·proof-first·Dual-Knowledge)을 실제 작동 코드로 옮겼어요. 다섯 결정 전부 공개 repo의 ③Gate에서 재현돼요.

04 — proof-first

“그럴듯한 화면”이 아니라, 실제로 플레이되는가

생성된 게임이 보기엔 멀쩡해도 실제론 멈춰 있는 경우가 많아요. AgentForge는 생성 전에 GDD에서 proof 타깃을 뽑고, 생성 뒤 headless로 도달 가능성을 검증해요. 통과 못 하면 배포되지 않아요.

검증 항목의미결과
playerMoves입력이 캐릭터를 실제로 움직이는가관측
scoreChanges플레이로 상태(점수)가 바뀌는가관측
win / fail 도달승·패 조건에 실제 도달 가능한가winReachable: true
dead / static 화면멈춘 게임을 통과시키는가pass: false 차단
예시 run: 20260629T050944Z-moon-fern-dash · reachability.json = pass:true · winReachable:true · [사실, 실재 — P3 ③Gate]
05 — 자기진화

지을수록 쌓이는 메모리

한 번 짜고 끝나는 게 아니라, 완성한 게임이 다음 게임의 재료가 돼요. 두 번째 생성이 첫 번째가 만든 템플릿을 실제로 재사용한 흔적(provenance)이 남고, 반복된 에러는 사전 체크 룰 후보로 쌓여요.

Template 재사용

혈통이 meta에 남는다

crystal-fox-dash에서 뽑은 대시 템플릿(stability 1)을 moon-fern-dash가 물려받아 재사용해요 → 이후 stability 2로 누적돼요. provenance는 meta.json의 usedTemplates.sourceRuns에 남아요.

Debug Generalizer

반복 에러 → 룰 후보

같은 에러 ≥3회 → 사전 체크 룰 후보를 만들어요. 검증된 fix만 기록해요(환각 0).

HITL 게이트

활성화는 사람이

룰 후보는 active: false · HITL 대기 상태예요. 작가 컨펌 전엔 적용되지 않아요. 자동 승격은 0이에요.

출처: 각 run meta.json(usedTemplates · evolution.templateExtracted) · studio-memory. 성숙의 흔적 — genesis인 crystal-fox는 winReachable:false(승리 도달 못 함)였지만, 그 템플릿을 물려받은 moon-fern은 winReachable:true. 단 “지을수록 쉬워지는” 누적 효과는 [측정전] — 메커니즘·코드는 검증, 효과는 게임이 더 쌓인 뒤 측정.

08 — 두 관점

어떤 직무로 읽히나

AI 응용 엔지니어 / FDE

프로덕션 에이전트 시스템

Codex CLI 멀티스텝 오케스트레이션 · proof-first 검증(dead 차단) · 자기진화(누적 메모리·HITL) · 비용/모델 인식. 에이전트를 “돌려보는” 게 아니라 운영하는 감각이에요.

게임 (게임 AI · 툴 · PM)

생산 시스템이 산출물

AI로 게임을 기획→에셋→코드→검증까지 만드는 시스템이에요. Game-PM Suite의 차별 멤버로, 다중관점 교차검증·proof-first 패턴이 Suite 전반과 맞닿아요.

09 — 정직 · 한계

증명한 것과, 아직 아닌 것

  • 측정전“지을수록 쉬워지는” 누적 효과는 아직 측정 전이에요. 메커니즘과 코드는 ③Gate로 검증됐지만, 생산성 효과는 게임이 N개 이상 쌓인 뒤에 측정해요.
  • 한계게임을 생성하려면 Codex CLI가 필요해서, 바로 브라우저로 데모하긴 어려워요(플레이는 브라우저에서 돼요). 멀티프레임 슬라이서·image_gen 1254px 고정·Chrome 의존은 알려진 제약이에요.
  • HITL룰 후보가 자동으로 승격되는 일은 없어요. 활성화는 항상 작가 컨펌을 거쳐요.
  • clean-room공개 repo는 ③Gate 클린이에요(시크릿 0). 외부 자산은 Codex CLI·Node/TS·React 같은 도구뿐이고, 모두 MIT 범위예요.
  • 범위vault game-studio-pipeline은 작가의 설계(provisional)이고, AgentForge는 그 이론을 독립적으로 구현한 거예요(같은 프로젝트는 아니에요). 사용자 평가는 아직 없어요(1인 프로토타입).
용어 풀이

비전문가용 한 줄

Codex CLI터미널에서 코드·이미지를 생성하는 AI 실행 도구예요. 여기선 Node가 단계별로 불러요.
image_genCodex에 내장된 이미지 생성이에요. 게임 스프라이트(픽셀 그림)를 만들어요.
headless화면 없이 브라우저를 돌려서, 게임이 진짜 작동하는지 자동으로 점검하는 방식이에요.
reachability(proof-first)“이 게임을 실제로 플레이해서 승리까지 갈 수 있나”를 만든 뒤에 증명하는 검증이에요.
GDDGame Design Document, 게임 기획서예요. 여기선 재미 규칙과 엔진이 실제로 할 수 있는 일을 함께 담아요.
HITLHuman-in-the-loop이에요. 자동화 중간에 사람 승인 단계를 끼워서, AI가 혼자 결정하지 않게 해요.

통제된, 그리고 진화하는 생산

AgentForge는 “AI가 게임을 만들어준다”가 아니에요. 만드는 과정을 들여다보고, 증명하고, 쌓아가는 시스템이에요. 코드는 공개 repo에서 ③Gate로 재현돼요.
